# Plan: 接入微信云开发联网房间流程
_Locked via grill - by Claude + 用户_

## Goal
在现有 Cocos Creator 3.8.8 大话骰项目里接入微信小游戏云开发，让联网模式跑通真实的获取头像昵称、分享、创房、加入、准备、摇骰、叫牌、开牌、结算、再来一局流程。联网状态由云函数权威判定，前端只负责授权、调用云函数、监听房间状态并渲染；单机模式保留但改成线下工具模式，只提供本机一副骰盅摇骰/看骰，不再模拟其他玩家。

## Approach
1. 将当前模型拆成两条清晰路径：联网房间继续使用扩展后的 `RoomService`/`RoomState`，单机工具使用独立的本地骰盅状态，不强塞进房间状态机。
2. 扩展联网玩家模型以支持微信头像、等待下局加入状态，并让 UI 能展示真实昵称和头像占位信息；第一版不承诺完整在线/观战语义，只区分 `self/active/waiting`，`PlayerId` 在联网模式下由云端 `OPENID` 映射生成，前端不得自报身份。
3. 新增微信配置文件，写入 `AppID=wx79916d4daf5d3ccd` 和 `envId=cloudbase-d3g1ll45q9cd4cc90`；前端启动联网模式时初始化 `wx.cloud`，非微信环境或配置缺失时给清晰提示。
4. 新增微信平台适配层，封装 `wx.getUserProfile` 或当前可用的授权入口、`wx.showShareMenu`、`wx.shareAppMessage`、`wx.onShareAppMessage`、`wx.getLaunchOptionsSync`、`wx.cloud.callFunction`、云数据库监听等能力，避免 `GameRoot.ts` 直接散落微信 API。
5. 调整 `GameRoot` 启动和入口流程：先根据分享参数或按钮选择服务工厂，再订阅对应服务；模式切换时必须清理旧订阅、旧状态、动画和按钮状态，避免联网入口继续跑本地服务。
6. 联网入口强制获取头像昵称：用户拒绝或失败时不创建/加入房间，提示需要授权后重试；成功后保存本次会话的昵称、头像 URL，并传给云函数。
7. 新增 `CloudRoomService` 真实现：`createRoom(profile)`、`joinRoom(roomId, profile)` 调用云函数；`ready`、`roll`、`bid`、`open`、`restart` 不再接受前端传入的 `playerId`，只传动作参数、`expectedVersion` 和一次性 `actionId`，身份由云函数从 `OPENID` 判定。
8. 新增云函数目录，例如 `cloudfunctions/room/index.js`，使用云开发 Node SDK 获取调用者 `OPENID`，并在云端执行所有房间动作、规则校验、骰子生成、结算统计和错误返回。
9. 设计云数据库集合：`rooms` 存公共房间投影，包括展示用短房号、高熵 `joinToken` 的哈希或等价校验值、阶段、房主 openid、当局玩家/等待玩家的公开信息、当前回合、叫牌历史、结算公开信息、版本号、创建/更新时间；未公开骰面不得直接放在客户端可监听的公共文档里。
10. 骰子保密：未结算骰面存到私有集合或按 openid 拆分的骰面文档，并只允许云函数读取；客户端监听 `rooms` 的公共投影，自己的骰面通过 `roll/getPrivateState` 类云函数返回，结算后云函数再把所有骰面写入公开结算投影。
11. 房间发现与读取权限：分享 query 同时带展示短房号和高熵 `joinToken`，客户端不能集合级列房或凭短房号订阅；必须先调用云函数校验 token，由云函数返回允许监听的单房间 doc id 或公共投影，数据库权限禁止客户端列表读取。
12. 所有云函数动作在事务内执行，校验 `phase/currentTurnPlayerId/lastBid/version/membership`，更新成功时递增 `version`；用 `actionId` 做幂等去重，处理双击、网络重试和多端同时操作。
13. 幂等记录裁剪：`actionId` 只保留最近 N 条，或写入带 `expireAt` 的独立幂等集合，避免长房间多局把 `rooms` 文档撑大。
14. 加入规则：房间最多 6 名当局玩家。按 `OPENID` 唯一识别身份，`join` 明确分成 active 重连、waiting 重连、新加入三种路径；如果房间处于 `lobby` 且 active 未满，新玩家进入当局；如果已经进入摇骰/叫牌/结算，新玩家进入等待下局列表并在 UI 显示“下局加入”；重开时把 waiting 合并进 active、重新分配空座位并保证 active <= 6。
15. 重开规则：只有 `phase === settlement` 可触发。第一版由房主点击“再来一局”重开，waiting 玩家不能触发；非房主 active 玩家看到等待房主重开。若后续要改成全员确认，另起需求。
16. 分享流程：创建房间后生成 `roomId` 和 `joinToken` query，注册右上角分享和顶部分享按钮；从分享进入时读取 query 并先完成头像昵称授权，再调用加入云函数校验 token。
17. 单机模式改为独立渲染分支和本地状态：只显示本机 5 个骰子、骰盅、摇骰按钮、上拖看牌/盖住，不显示房间号、玩家列表、分享、准备、叫牌、开牌、结算和模拟 AI。
18. 保留当前本地多人逻辑作为可选参考或删除入口，但不要再让“单机模式”进入 4 人模拟房；联网模式使用云服务。
19. 同步更新 `tools/generate-scene.mjs` 和 `assets/main.scene`：所有新增授权、重试、等待下局、联网错误、单机工具控件都必须预生成；动态玩家相关节点要采用稳定模板/安全命名策略，避免运行时 `createNode` 找不到节点而崩溃。
20. 更新 README，写清微信开发者工具配置、云开发环境 ID、数据库集合、云函数部署、权限建议、预览验证步骤。
21. 验证：补充可运行的云函数核心规则测试脚本，覆盖创房、加入/重连、等待下局合并、重复 actionId、非法回合动作、骰子可见性、token 校验、房主重开权限和结算；执行 `npm run generate:scene` 验证场景节点；若当前 Cocos 类型检查仍依赖编辑器生成的 `temp/tsconfig.cocos.json`，README 中列出可执行的开发者工具构建/预览检查。

## Key decisions & tradeoffs
- 联网模式采用云函数权威判定，而不是前端直连数据库改状态；代价是要新增云函数和部署步骤，收益是并发、骰子保密、防作弊和规则一致性更稳。
- 头像昵称授权是硬门槛；拒绝时不允许联网创房/加入，只提供重试。这牺牲低摩擦进入，但符合用户对“获取人信息”的明确要求。
- 开局后新玩家允许加入但只等到下一局参与；这让分享随时有效，同时避免中途插入影响当前回合和骰子总数。
- 分享入口使用短房号加高熵 `joinToken`，而不是只靠短房号；代价是 query 更长，收益是降低枚举房间和旁观读取风险。
- 第一版重开由房主在结算阶段触发；这比全员确认简单，但可能让房主更快推进流程。
- 单机模式重新定义为线下“一副骰盅”工具，不再复用本地 4 人模拟房；这会移除当前单机完整游戏闭环，但更符合线下两人纯玩的使用场景。
- 第一版只做一个房间云函数和少量集合，便于落地和部署；但公共房间投影与私有骰面必须分离，不能为了简单把未公开骰面暴露给监听端。
- 云函数动作接口不信任客户端传来的玩家身份；代价是前端接口要重构，收益是杜绝伪造其他玩家行动。

## Risks / open questions
- 微信小游戏当前获取头像昵称 API 可能受基础库和审核规则影响；实现时要把授权入口封装好，并在 README 中注明如果 `getUserProfile` 在目标基础库不可用，需要切换到微信推荐的新头像昵称填写能力。
- Cocos Creator 构建到微信小游戏后的云函数目录是否自动纳入发布，取决于项目配置和微信开发者工具操作；代码会提供目录和说明，但部署需要用户在开发者工具完成。
- 云数据库权限需要谨慎：房间写操作应主要通过云函数，客户端监听只读；若权限配置过宽会削弱云函数权威性。
- 多人同时叫牌/开牌可能出现并发竞争；第一版需要云端事务、版本号和 `actionId` 幂等，不能只靠前端禁用按钮。
- 远程头像 URL 在 Cocos 里渲染可能受跨域、加载器和微信小游戏纹理限制影响；第一版可以先展示昵称和默认头像色块，头像贴图加载作为增强。
- 微信云开发数据库监听权限可能与私有骰面拆分冲突；实现时优先保证不泄密，必要时牺牲实时性，用动作返回和公共投影刷新同步个人骰面状态。
- 第一版不做完整在线/离线判断；UI 不应展示会被误解为实时在线状态的文案。

## Out of scope
- 正式运营级防作弊、心跳在线、断线托管、房主转移、踢人、观战聊天。
- 积分、扣骰、淘汰、战绩、排行榜和账号长期资料。
- 云函数自动部署脚本、CI/CD、生产环境监控和告警。
- 中途加入当前局、旁观视角完整功能、同房间多桌。
- 精修 UI 美术或重新设计整体界面。
