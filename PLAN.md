# 计划：Cocos Creator 大话骰微信小游戏原型
_已通过 grill 锁定 - Claude + 用户_

## 目标
基于 Cocos Creator 3.8.8 开发一个竖屏 2D 大话骰微信小游戏原型。第一版先跑通本地模拟房间流程：创建房间、分享/加入入口、本地模拟玩家、准备、摇骰、叫牌、开牌、结算、再来一局。缺失的骰子、骰盅等素材用生成式图片补齐为透明 PNG。真实微信云开发暂不实现，但代码需要保留房间服务接口边界，后续可以把本地模拟服务替换成云开发同步服务。

## 实施方案
1. 以当前磁盘工程为准，使用 Cocos Creator 3.8.8。由于当前 `assets` 目录为空，重新建立可维护的 `assets` 资源与脚本结构。
2. 生成缺失的项目素材，风格为“国风棋牌室 + 轻写实 2D”：骰子、骰盅、按钮/面板、桌面/背景等必要 PNG。需要透明底的素材先生成纯色抠图背景，再本地去底，验证 alpha 通道后保存到 `assets`。
3. 创建一个很薄的启动场景/入口组件，主要 UI 在 TypeScript 里运行时动态创建，不手写复杂 Cocos 场景 JSON。
4. UI 按竖屏 `720 x 1280` 设计：顶部为房间号、分享、状态；中部为桌面和其他玩家位；底部为本机玩家骰子/骰盅与操作区。
5. 定义游戏数据模型：玩家、房间状态、骰子点数、叫牌、结算结果、游戏阶段。
6. 定义 `RoomService` 接口，覆盖创建房间、加入房间、分享数据、准备、摇骰、叫牌、开牌、再来一局、监听房间状态。
7. 实现 `LocalRoomService` 作为第一版可玩服务。默认创建 4 人房，逻辑支持 2-6 人，每人 5 个骰子；非本机玩家自动准备、摇骰，并在轮到他们时自动合法叫牌或开牌。
8. 增加 `CloudRoomService` 空壳，作为后续微信云开发适配层。第一版不实现真实云集合、云函数、权限规则或网络同步。
9. 实现游戏规则：所有玩家准备后开始；每人每轮只能摇一次；所有人摇完后开始叫牌；第一局从房主开始叫，后续从上一局输家开始叫；下一手叫牌必须数量更大，或数量相同但点数更大；`1` 默认是万能点，但只要本轮有人叫过 `1`，之后 `1` 只算自己。
10. 实现开牌与结算：开牌后显示所有骰子，统计 1-6 每个点数数量，按万能 `1` 规则判断上一手叫牌是否成立，计算输家，展示结算面板，并提供“再来一局”。
11. “再来一局”只重置单局状态：玩家保留，准备/摇骰/骰子/叫牌重置，下一局由输家先叫。不做扣骰、积分、淘汰或多局赛制。
12. 验证标准是整体流程跑通：创建房间 -> 模拟玩家加入/准备 -> 开始 -> 每人摇一次 -> 轮流叫牌/开牌 -> 结算 -> 再来一局。如果当前环境无法启动 Cocos Creator，则至少做 TypeScript 静态检查和文件结构检查，并明确需要在编辑器里手动预览验证。

## 关键决策与取舍
- 使用 Cocos Creator 3.8.8，而不是 3.3.8，因为当前工程配置写的是 3.8.8，且用户已确认。
- 采用运行时动态创建 UI，而不是直接编辑复杂场景 JSON，因为当前 `assets` 为空，动态 UI 更稳定，也避免 Cocos 序列化和 UUID 绑定问题。
- 第一版做本地模拟，保留云开发接口边界，因为真实微信云开发会引入后端数据结构、权限、安全、并发、防作弊等额外范围。
- 支持 2-6 人、每人 5 骰，本地默认 4 人房，方便单人测试完整流程。
- 采用常见大话骰规则：`1` 默认万能；叫过 `1` 后，本轮 `1` 不再万能。
- 第一版只做单局闭环和再来一局，不做扣骰、积分、淘汰。
- 缺失美术先用生成素材补齐，并保持资源路径稳定，后续正式美术可替换。

## 风险与未决事项
- 当前 `assets` 目录为空，只有 `library` 缓存文件，可能没有可靠的原始场景或原始美术资源可复用。
- 程序生成最小 Cocos 场景可能仍需要 Cocos Creator 编辑器导入/刷新 UUID；实现时应尽量使用简单、编辑器友好的文件结构，不依赖 `library` 缓存。
- 透明 PNG 素材如果抠图边缘有残留，可能需要一次迭代修正。
- 如果当前环境无法启动 Cocos Creator，验证会受限于静态检查和文件结构检查，最终预览需要用户在编辑器里确认。
- 后续真实 `CloudRoomService` 需要处理权威状态、并发操作、骰子保密、断线重连、房间清理、防作弊等问题；这些不属于本轮原型范围。

## 不包含范围
- 真实微信云开发实现、云函数、数据库集合、权限规则、环境 ID 配置和生产级同步。
- 真实多人网络、断线重连、房主转移、防作弊。
- 积分、扣骰、淘汰、排位或长期成长系统。
- 最终精修美术；本轮只生成可用的原型 PNG 素材。
- 手动编辑器 UI 布局或完整 prefab 工作流。
